## 🔧 一、总体开发约定

| 项          | 说明                                                   |
| ----------- | ------------------------------------------------------ |
| 项目架构    | 简化三层架构：Controller + Service 实现类 + Repository |
| ORM 框架    | Spring Data JPA（基于 Hibernate）                      |
| JSON 序列化 | 默认 Jackson                                           |
| DTO 映射    | 不使用 `BeanUtils.copyProperties`                      |
| 接口风格    | 统一使用 `ResponseEntity<?>` 包装 JSON 格式响应        |
| 分页查询    | 使用 `Pageable` 和 `Page<T>`                           |

------

## 🧩 二、包结构规范

```text
com.xxx.project
├── controller          # 控制器层
├── service             # 业务实现层（无接口）
├── repository          # DAO 层
├── model
│   ├── entity          # 实体类（与数据库字段映射）
│   ├── dto
│   │   ├── request     # 接收前端请求参数
│   │   └── response    # 返回前端的格式化对象
├── config              # 配置类
├── util                # 工具类
```

## 🧭 三、命名规范

### 1. Java 实体类字段名

- 使用驼峰命名（如：`userId`, `createTime`）
- 加上 `@Column(name="xxx")` 映射数据库字段

### 2. 接口路径

- RESTful 风格 + 动词，如：
  - `POST /api/user/register`
  - `GET /api/admin/getAllUsers`

------

## 🧪 四、控制器层规范

- 使用 `@RestController` + `@RequestMapping("/api/xxx")`

- 所有接口返回值统一使用：

  ```java
  return ResponseEntity.ok(Map.of(
      "code", 200,
      "message", "xxx成功",
      "data", xxx
  ));
  ```

- 失败返回：

  ```java
  return ResponseEntity.badRequest().body(Map.of(
      "code", 400,
      "message", "xxx失败：" + e.getMessage()
  ));
  ```

- 请求参数统一封装为 DTO 请求类（如：`UserRequest`）

------

## 🧱 五、Service 层规范

- 不定义接口，直接使用 `@Service` 的实现类
- 所有数据库逻辑、业务校验都写在这里
- 异常统一抛出 `RuntimeException`，Controller 统一处理返回

------

## 📤 六、Request/Response 规范

### 请求 DTO（Request）

```java
@Data
public class UserRequest {
    private Integer page;
    private Integer size;
    private String name;
}
```

- 命名为 `XXXRequest`，放在 `dto.request` 包中

### 响应 DTO（Response）

```java
@Data
@AllArgsConstructor
@NoArgsConstructor
public class UserResponse {
    private Integer id;
    private String name;
    private String role; // 中文：超级管理员、普通用户
    private String state; // 中文：新建、已激活、已删除
}
```

- 命名为 `XXXResponse`，放在 `dto.response` 包中
- 与 Entity 分离，避免暴露数据库结构

## 🔁 七、字段转换规范

- 统一在 Service 层中进行 DTO → Entity 映射

- 使用：

  ```java
  UserResponse dto = new UserResponse();
  dto.setId(user.getId());
  dto.setAccount(user.getAccount());
  dto.setUsername(user.getUsername());
  dto.setPhone(user.getPhone());
  dto.setCreateTime(user.getCreateTime());
  dto.setRole(roleToCn(user.getRole()));
  dto.setState(stateToCn(user.getState()));
  ```

  ## 🧱 实体和响应类映射规范

  - 所有 `Entity → DTO` 的转换都采用**手动赋值**
  - 建议封装在 `Service` 层的私有方法中统一处理
  - 如果字段多，可使用 IDEA 的快捷键 `Alt + Insert → Getter/Setter`

  ### 示例：统一映射方法

  ```java
  private UserResponse toUserResponse(User user) {
      UserResponse dto = new UserResponse();
      dto.setId(user.getId());
      dto.setAccount(user.getAccount());
      dto.setUsername(user.getUsername());
      dto.setPhone(user.getPhone());
      dto.setCreateTime(user.getCreateTime());
      dto.setRole(roleToCn(user.getRole()));
      dto.setState(stateToCn(user.getState()));
      return dto;
  }
  ```

##  八、异常处理与校验规范（统一使用 `BusinessException`）

### ✅ 处理目标

- 所有业务异常、参数错误统一抛出 `BusinessException`
- Controller 层统一捕获 `BusinessException`，并返回前端友好的 JSON 响应
- 不再使用原生 `RuntimeException` 或 `IllegalArgumentException`

------

### 📦 自定义异常类示例

```
package com.wang.fireserver.exception;

public class BusinessException extends RuntimeException {
    private final int code;

    public BusinessException(String message) {
        super(message);
        this.code = 400;
    }

    public BusinessException(int code, String message) {
        super(message);
        this.code = code;
    }

    public int getCode() {
        return code;
    }
}
```

------

### 🛠️ Controller 层统一异常处理（建议使用 `@ControllerAdvice`）

```
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<?> handleBusinessException(BusinessException ex) {
        return ResponseEntity.badRequest().body(Map.of(
                "code", ex.getCode(),
                "message", ex.getMessage()
        ));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<?> handleOtherExceptions(Exception ex) {
        return ResponseEntity.status(500).body(Map.of(
                "code", 500,
                "message", "服务器内部错误: " + ex.getMessage()
        ));
    }
}
```

------

### ✍️ Service 层抛出示例

```
@Transactional
public void deleteUser(List<Integer> userIds) {
    if (userIds == null || userIds.isEmpty()) {
        throw new BusinessException("用户 ID 列表不能为空");
    }

    for (Integer id : userIds) {
        User user = userRepository.findById(id)
                .orElseThrow(() -> new BusinessException("用户不存在: " + id));
        user.setState("X"); // 逻辑删除
        userRepository.save(user);
    }
}
```

## 📦 九、Excel 导出规范

- 所有导出逻辑写在 Service 层
- Controller 只调用导出方法，示例：

```java
@PostMapping("/exportUsers")
public void exportUsers(HttpServletResponse response) throws IOException {
    String requester = jwtUtil.getAccountFromToken(); // 自定义方法
    if (!userService.isAdmin(requester)) {
        throw new RuntimeException("无权限");
    }
    userService.exportUsers(response);
}
```

## ✅ 总结关键词（方便 AI 识别你的习惯）

- Spring Boot + JPA + Lombok 项目结构
- 不使用接口的 Service 层
- Controller 只做参数收取和 ResponseEntity 封装
- Service 负责所有业务逻辑
- BeanUtils 处理字段映射
- JSON 请求使用 DTO Request 对象
- JSON 响应统一包装为 Map（status/message/data）
- 分页查询用 PageRequest + PageImpl
- Excel 导出逻辑集中写在 service 层
